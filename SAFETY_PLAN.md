# Safety Plan

**Project:** rust-CAN

**Standard:** ISO 26262:2018 (Road vehicles — Functional safety)  
**ASIL Target:** ASIL-B (all safety goals)  
**Date:** 2026-06-18  

---

## 1. Purpose and Scope

This Safety Plan governs the safety activities for the rust-CAN library, a Rust-language CAN bus abstraction conformant to RELAY spec v1.1.

rust-CAN provides:
- CAN, CAN FD, and CAN XL frame handling
- Frame validation per RELAY §15.1
- E2E (End-to-End) CRC-16 protection for safety-critical payloads
- ISO-TP (ISO 15765-2) transport layer
- J1939 protocol support
- DBC signal decoding
- SocketCAN (Linux AF_CAN) and virtual bus implementations
- RELAY protocol adapter

## 2. Safety Goals

| ID         | Description                                             | ASIL   |
|------------|---------------------------------------------------------|--------|
| SG-CAN-01  | Reject frames violating CAN protocol constraints        | ASIL-B |
| SG-CAN-02  | Detect payload corruption and replay attacks via E2E    | ASIL-B |
| SG-CAN-03  | Bound all blocking receives by caller-supplied deadline | ASIL-A |

Full HARA: see `.fusa-hara.json`.

## 3. Requirements

High-level safety requirements are maintained in `.fusa-reqs.json` in rsfusa format (36 requirements, REQ-CAN-001 through REQ-DBC-002). Every requirement is:

- Tagged with `"standard": "iso26262"` and `"level": "HLR"`
- Annotated in source with `//fusa:req REQ-XXX-NNN`
- Traced to at least one test with `//fusa:req` above `#[test]`

## 4. Development Process

### 4.1 Coding Standard
See `CODING_STANDARD.md` — derived from ISO 26262 Part 6 Table 1.

Key rules:
- `#![forbid(unsafe_code)]` at crate root
- All functions ≤ 60 statements (enforced by rsfusa comp V(G) ≤ 10)
- No unwrap/expect in non-test code
- Every safety-relevant function annotated with `//fusa:req`

### 4.2 Verification

| Activity               | Tool / Method              | Exit Criterion                        |
|------------------------|---------------------------|---------------------------------------|
| Unit tests             | `cargo test`               | 0 failures                            |
| Integration tests      | `cargo test --tests`       | 0 failures                            |
| Static analysis        | `rsfusa analyze`           | 0 ERROR-level findings                |
| Coding standard        | `rsfusa lint`              | 0 violations                          |
| Complexity             | `rsfusa comp`              | V(G) ≤ 10 per function                |
| Requirements trace     | `rsfusa trace`             | 100% HLR requirements traced to test  |
| Safety check           | `rsfusa check --strict`    | 0 ERRORs                              |
| FMEA                   | `fmea.json`                | All ASIL-B items mitigated            |
| TARA                   | `tara.json`                | All SL-2 threats mitigated            |

### 4.3 Continuous Integration

Every PR must pass:
1. `cargo fmt --check` — formatting
2. `cargo clippy -D warnings` — linting
3. `cargo test --locked` — all tests
4. ASIL-B safety job (rsfusa full pipeline)
5. DCO sign-off check
6. Cross-compile (aarch64/x86_64 musl)

## 5. Tool Qualification

rust-FuSa (rsfusa) is used as a safety analysis and verification tool. Tool qualification evidence is produced by `rsfusa qualify` on each release and archived in the audit pack.

The Rust compiler and cargo are qualified via:
- Version pinning in `Cargo.lock` and `rust-toolchain.toml`
- Verified against known compiler test suite results

## 8. Known Limitations

1. SocketCAN support is Linux-only (`cfg(target_os = "linux")`). macOS/Windows use VirtualBus only.
2. CAN XL hardware support depends on kernel ≥ 6.0 with XL-capable drivers.
3. ISO-TP multi-frame receive has no ASIL assignment (QM) until timeout is fully validated.
